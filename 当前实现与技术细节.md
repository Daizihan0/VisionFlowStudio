# VisionFlow Studio 当前实现与技术细节

## 1. 当前版本范围

当前仓库已经完成一版可运行的 `Windows x64` 桌面自动化 MVP，包含：

- 节点连线式流程设计器
- 项目保存 / 加载（JSON）
- 截图定位
- 剪贴板贴图定位
- 手动十字定位
- OpenCV 图像模板匹配
- 像素颜色判断
- 鼠标模拟：单击 / 双击 / 右键 / 中键 / 滚轮
- 键盘模拟：文本输入 / 快捷键 / 虚拟键回放
- 等待节点 / 条件节点 / 分支执行
- 录制片段节点：全局录制 + 回放
- 全局热键：`F9` 停止录制，`F10` 紧急停止
- 自包含发布脚本

当前目标仍然是：**普通办公软件自动化**，不以游戏、反作弊环境、驱动级注入场景为目标。

## 4. 视觉能力实现细节

### 4.1 截图定位

实现方式：

- `ScreenCaptureService` 先最小化主窗口
- 通过 `Graphics.CopyFromScreen` 截取整个虚拟桌面
- 通过 `SnippingOverlayWindow` 全屏覆盖层让用户框选区域
- 框选结果转成 `BitmapSource`
- 最后转成 PNG Base64，写入当前节点的 `AssetPayloadBase64`

当前存储方式：

- 模板图直接内嵌在节点 JSON 中
- 优点：项目单文件可带走
- 缺点：工程文件体积会快速膨胀

### 4.2 粘贴图片定位

实现方式：

- 直接读取系统剪贴板中的图像
- 转换为 PNG Base64
- 绑定到当前节点

### 4.3 手动十字定位

实现方式：

- 截取整个虚拟桌面
- 使用 `CrosshairPickerWindow` 覆盖层显示十字线
- 用户点击任意像素点后记录：
  - 绝对屏幕坐标
  - 当前像素颜色（十六进制）

自动写回当前节点参数：

- `Point=x,y`
- `PixelColor=#RRGGBB`
- `Tolerance=12`

并根据节点类型自动调整：

- 动作节点：`TargetMode=AbsolutePoint`
- 条件节点：`ConditionType=PixelEquals`
- 等待节点：`WaitType=PixelEquals`

### 4.4 图像匹配

实现方式：

- 使用 `OpenCvSharp`
- 将屏幕截图和模板图转换为 `Mat`
- 若是 4 通道图像先转 `BGR`
- 调用 `Cv2.MatchTemplate(..., CCoeffNormed)`
- 用 `MinMaxLoc` 取最高匹配分数
- 分数达到阈值则认为匹配成功

当前阈值来源：

- 节点参数 `Threshold`
- 默认 `0.92`

当前限制：

- 只做了基础模板匹配
- 还没有做灰度匹配开关
- 还没有做局部区域限制
- 还没有做缩放自适应
- 还没有做多模板回退

## 5. 输入模拟实现细节

### 5.1 鼠标模拟

使用：

- Win32 `SendInput`
- 绝对坐标 + `MOUSEEVENTF_ABSOLUTE`
- 同时开启 `MOUSEEVENTF_VIRTUALDESK`

已实现动作：

- 左键单击
- 左键双击
- 右键
- 中键
- 滚轮上下
- 鼠标移动到绝对位置

### 5.2 键盘模拟

已实现两种模式：

1. Unicode 文本输入
   - 适合普通文本输入
2. 虚拟键输入
   - 适合快捷键与录制回放

当前可用快捷键例子：

- `Ctrl+V`
- `Ctrl+C`
- `Alt+Tab`
- `F1` ~ `F24`

### 5.3 当前输入模拟限制

以下情况后续要特别注意：

- 管理员权限窗口：若主程序不是管理员，输入可能失效
- 某些安全软件/银行软件/游戏窗口会拦截 `SendInput`
- 某些 IME/富文本控件对 Unicode 注入响应不一致
- 多显示器 + DPI 缩放下，极少数边界点仍可能出现偏差

## 6. 录制与回放实现细节

### 6.1 录制实现

使用：

- Win32 全局低级钩子
  - `WH_KEYBOARD_LL`
  - `WH_MOUSE_LL`

录制内容：

- 键盘按下
- 键盘抬起
- 鼠标左/右/中键弹起
- 鼠标滚轮

记录字段：

- 事件类型
- 距上一个事件的延迟时间 `DelayMs`
- 鼠标坐标
- 鼠标按键
- 滚轮增量
- 虚拟键码

### 6.2 当前录制策略

当前是“原始事件录制”，优点是简单直接，缺点也明显：

- 事件数量可能较多
- 回放对界面变化不敏感
- 纯坐标回放稳定性一般

### 6.3 回放实现

- `SubFlow` 节点读取 `RecordedEvents`
- 顺序执行每个事件
- 使用记录下来的 `DelayMs` 保留节奏
- 鼠标事件走 `SendInput`
- 键盘事件走虚拟键发送

### 6.4 当前热键

- `F9`：停止录制
- `F10`：紧急停止当前执行；如果正在录制，也会先停录

## 7. 真实执行引擎设计

`DesktopFlowExecutionEngine` 当前支持：

### 7.1 `Vision` 节点

- 读取节点模板图
- 按阈值在全屏查找
- 成功后把中心点写入运行态锚点

### 7.2 `Action` 节点

支持：

- `LeftClick`
- `DoubleClick`
- `RightClick`
- `MiddleClick`
- `WheelUp`
- `WheelDown`
- `TypeText`
- `Hotkey`

定位来源：

- `AbsolutePoint`
- 上一个视觉锚点
- 视觉锚点偏移

### 7.3 `Wait` 节点

支持：

- 固定延时
- 等待像素匹配
- 等待图片出现
- 等待图片消失

### 7.4 `Condition` 节点

支持：

- 图片存在判断
- 像素判断

### 7.5 `SubFlow` 节点

支持：

- 回放当前节点中的录制事件序列

### 7.6 流程跳转

当前规则：

- `Condition` 节点根据结果走 `True / False`
- 其余节点优先走 `Success`，否则走 `Next`
- 失败分支仅支持显式 `Failure` 连线

## 9. 后续继续开发时最容易犯的错

下面这些是后续跟进时**特别容易踩坑**的地方，建议每次改动都对照检查。

### 9.1 不要把设计器状态和运行时状态混写

容易犯的错：

- 运行过程中直接改节点结构
- 把临时锚点、匹配结果写回长期配置

建议：

- 设计态数据放 `FlowNode`
- 运行态瞬时数据放独立 `RuntimeState`
- 不要让执行器直接修改图结构

### 9.2 不要继续把所有截图都内嵌 Base64 到 JSON 里

当前 MVP 为了单文件工程方便这么做，但后面会出问题：

- 工程文件变得很大
- Git diff 难看
- 存档和合并都不友好

建议后续改成：

- 工程 JSON 只记录资源路径/资源 ID
- 实际模板图放到项目目录的 `assets/images/`

### 9.3 录制回放不能长期停留在“原始事件回放”阶段

容易犯的错：

- 误以为录制功能已经稳定
- 把所有流程都建立在绝对坐标回放上

风险：

- 窗口位置变化即失效
- 控件移动或 UI 变化即失效
- 时间差稍变就会点错位置

建议后续做：

- 录制后清洗事件
- 自动合并无效键抬起/重复点击
- 支持把录制中的关键点击替换为视觉锚点点击

### 9.4 条件分支自动连线逻辑现在很简化

当前自动连线规则很轻量，后续如果继续堆功能，容易出现：

- 一个条件节点被错误追加多个相同出口
- 用户拖拽重排后连线语义不清

建议后续做：

- 正式的端口系统
- 输入/输出端口显式化
- 节点校验器

### 9.5 全局钩子与热键一定要保证释放

非常容易犯的错：

- 程序异常退出后钩子未释放
- 窗口关闭时忘记注销热键
- 录制状态切换时重复挂钩

建议：

- 所有钩子都走统一生命周期管理
- `Dispose` 路径保持幂等
- 异常时也要走释放逻辑

### 9.6 `SendInput` 的线程与节奏不要乱改

容易犯的错：

- 为了“更快”把所有 `Task.Delay` 都删掉
- 并发执行多个输入动作

风险：

- 点击顺序紊乱
- 快捷键按下抬起顺序错乱
- 目标程序未响应导致失败率飙升

建议：

- 保持输入串行
- 把节奏抽成可配置，但必须有下限
- 不要在多个线程同时发输入

### 9.7 高 DPI 和虚拟桌面坐标不要混淆

容易犯的错：

- 逻辑坐标和物理像素混用
- 只在主屏上测试

建议：

- 后续把所有坐标类型显式建模
- 区分屏幕绝对坐标、窗口客户区坐标、图像匹配区域坐标
- 必须做双屏、高分屏回归测试

### 9.8 不要把 `Wait` / `Condition` 再继续硬编码扩散

当前通过 `Settings[key]` 快速推进了 MVP，但后面继续堆 if/else 很快会不可维护。

建议：

- 后续为节点配置引入强类型配置对象
- 至少为 `ActionConfig`、`VisionConfig`、`WaitConfig`、`ConditionConfig` 分层

### 9.9 当前 WPF 设计器还不是真正的专业流程编辑器

容易犯的错：

- 直接在现在的 `Canvas + ItemsControl` 上无限加复杂编辑功能

风险：

- 连线端口管理越来越乱
- 命中测试、缩放、框选、多选会越来越难维护

建议后续两种路线选一：

1. 继续自研，但尽早抽出 `DesignerCanvas / NodeControl / ConnectionControl`
2. 评估是否引入成熟的 WPF 图编辑基础库

### 9.10 构建产物不要提交进 Git

当前已用 `.gitignore` 排除了：

- `bin/`
- `obj/`
- `artifacts/`

后续不要把这些目录重新纳入版本控制，否则仓库会迅速膨胀。

## 10. 下一阶段建议优先级

我建议后续开发顺序如下：

1. 录制结果清洗与事件编辑器
2. 资源外置化（图片从 Base64 改为 assets 文件）
3. 窗口锚定 / 窗口查找
4. 节点端口化与更稳定的连线编辑
5. 定时执行真正落地
6. 失败截图、断点、单步调试
7. OCR 预留接口

## 11. 当前版本总结

这版已经从“纯设计器骨架”推进到“可真实操作桌面”的 MVP：

- 能设计流程
- 能截图找图
- 能按像素/图像等待与分支
- 能真正点击与输入
- 能录制并回放一段操作
- 能通过热键强行中断

但它仍然是 **MVP**，不是最终稳定商业版。后续最关键的提升方向不是继续堆功能，而是：

- 稳定性
- 资源管理
- 配置强类型化
- 录制结果清洗
- 设计器可维护性
